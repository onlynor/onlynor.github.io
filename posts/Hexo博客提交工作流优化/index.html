Hexo 博客提交工作流优化

背景

使用 Hexo + Fluid 主题搭建博客后，最初的部署方案是 hexo deploy，通过 hexo-deployer-git 插件自动推送生成文件到 GitHub Pages。

后来改为手动管理 public/ 目录的 git 仓库，直接推送静态文件。这个过程中踩了不少坑，最终形成了一套更稳定的提交方案。

最初的问题：hexo deploy 的局限

hexo deploy 本质是把 public/ 的内容推送到远程仓库。它能用，但有几个问题：

  • 每次部署都要重新构建并推送整个站点
  • 推送过程不透明，出问题不好排查
  • token 配置在 _config.yml 里，有安全风险

改进方案：手动管理 public 的 git

把 git 仓库直接放在 public/ 目录里，手动控制提交和推送：

这样做的好处：

  • 完全掌控推送内容和时机
  • 可以查看 diff，确认改了什么
  • 不需要在 _config.yml 里配置 token

核心优化：用 hexo g –force 替代 hexo clean

最初的习惯是 hexo clean && hexo g，但 hexo clean 会删除整个 public/ 目录，包括里面的 .git。每次 clean 之后都要重新 git init 和 git remote add，非常麻烦。

看了 Hexo 源码确认，hexo clean 调用的是 rmdir，直接递归删除整个 public 文件夹：

而 hexo g --force 会强制重新生成所有文件，但不会删除目录本身，.git 完好无损。

所以最终方案是：永远不用 hexo clean，用 hexo g --force 代替。

踩过的坑

Steam++ 劫持 SSH

Steam++ 会修改 hosts 文件，把 github.com 指向 127.0.0.1。浏览器通过本地代理能正常访问 GitHub，但 SSH 端口 22 没人监听，导致 git push 报错：

解决：推送前关闭 Steam++，或关闭其 GitHub 加速功能。

_config.yml 不提交

_config.yml 里有 GitHub token，不能提交到公开仓库。把它加入了 .gitignore，但换电脑时需要手动恢复配置。

最终方案总结

操作 命令
生成静态文件 hexo g --force
推送到 GitHub cd public && git add -A && git commit -m "update" && git push
本地预览 hexo s
新建文章 hexo new "文章标题"

不用 hexo deploy，不用 hexo clean，简单可控。
